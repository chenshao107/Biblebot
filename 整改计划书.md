# 企业内部知识探索 Agent 实现方案（初版）

## 一、目标

构建一个：

```text
企业内部知识探索 Agent 系统
```

具备：

* 企业知识库问答
* 文档探索
* 代码探索
* 多轮 Agent 推理
* MCP 工具调用
* Bash/CLI 探索能力
* WebUI 对话
* 内部 RAG 检索

并且：

```text
Agent 本身尽量无状态、只读、安全、可替换
```

核心理念：

```text
轻RAG + 强探索
```

而不是：

```text
一次性塞大量上下文
```

---

# 二、整体架构

```text
┌────────────────────────────┐
│ Host                       │
│ ingestion pipeline         │
│                            │
│ - 文档解析                 │
│ - md转换                   │
│ - chunk                    │
│ - embedding                │
│ - 写入Qdrant               │
└────────────┬───────────────┘
             │
             ▼
┌────────────────────────────┐
│ Qdrant Docker              │
│                            │
│ 向量数据库                 │
│ 仅负责存储与检索           │
└────────────┬───────────────┘
             │
             ▼
┌────────────────────────────┐
│ Agent Docker               │
│                            │
│ - Open WebUI               │
│ - Agent Runtime            │
│ - MCP Client               │
│ - CLI Tools                │
│ - rag_search               │
│                            │
│ 负责：                     │
│ reasoning / orchestration  │
└────────────────────────────┘
```

---

# 三、职责划分

# 1. Host（离线数据准备）

负责：

* 扫描知识库目录
* markdown/pdf/doc/docx 转换
* chunk
* embedding
* 增量更新
* 写入 Qdrant

特点：

```text
可写
工程化
离线
```

不属于 Agent Runtime。

---

# 2. Qdrant Docker

使用：

[Qdrant](https://qdrant.tech/?utm_source=chatgpt.com)

职责：

* vector storage
* semantic retrieval
* payload filter

不负责：

* chunk
* rerank
* prompt
* agent logic

特点：

```text
状态持久化
独立服务
```

---

# 3. Agent Docker

职责：

* Agent 推理
* Tool orchestration
* MCP 调用
* Bash 探索
* 文档阅读
* WebUI 对话

特点：

```text
尽量无状态
尽量只读
可随时重建
```

---

# 四、推荐 Docker 结构

推荐：

```text
docker-compose
├── agent
└── qdrant
```

数据准备：

```text
host ingestion pipeline
```

不建议现阶段 Docker 化 ingestion。

---

# 五、推荐 Docker Compose

```yaml
version: "3"

services:

  qdrant:
    image: qdrant/qdrant
    volumes:
      - ./qdrant_data:/qdrant/storage
    expose:
      - "6333"

  agent:
    build: ./agent
    depends_on:
      - qdrant

    volumes:
      - ./knowledge:/knowledge:ro

    ports:
      - "3000:3000"
```

---

# 六、知识库目录挂载原则

Agent Docker：

```text
readonly mount
```

例如：

```yaml
volumes:
  - ./knowledge:/knowledge:ro
```

目的：

```text
Agent 只能探索知识
不能修改知识源
```

---

# 七、RAG 设计原则

# 核心思想

```text
RAG 负责“定位”
Agent 负责“探索”
```

不要：

```text
一次返回大量全文
```

推荐：

```json
[
  {
    "path": "...",
    "title": "...",
    "score": 0.91,
    "snippet": "..."
  }
]
```

然后：

Agent 自行：

* cat
* grep
* refine search

---

# 八、推荐 Retrieval 工作流

```text
rag_search
    ↓
返回候选文档
    ↓
Agent 自主阅读
    ↓
继续 refine search
    ↓
最终生成答案
```

---

# 九、为什么不让 Agent 直连 Qdrant

原因：

Qdrant 是：

```text
storage engine
```

而不是：

```text
Agent-friendly retrieval layer
```

Agent 真正需要的是：

```text
rag_search(query)
```

而不是：

```text
vector db 操作
```

---

# 十、rag_search 设计（重点）

推荐：

```bash
rag_search "蓝牙断连"
```

内部：

```text
query rewrite
→ embedding
→ qdrant
→ rerank
→ snippet extraction
```

最后返回：

```json
[
  {
    "path": "/docs/bluetooth/pair.md",
    "score": 0.92,
    "snippet": "..."
  }
]
```

---

# 十一、Tool 设计原则

# MCP 适合：

结构化、高语义业务系统。

例如：

* Phabricator
* Jira
* Review系统
* Issue系统

---

# CLI Tool 适合：

探索型工具。

例如：

* rag_search
* web_search
* grep_docs
* repo_search
* log_search

---

# 十二、为什么 CLI 更适合探索

Claude/OpenHands 天然擅长：

```bash
find
grep
cat
head
tail
```

形成：

```text
search → read → refine
```

工作流。

---

# 十三、CLI Tool 规范（重要）

不要只提供：

```bash
--help
```

而是：

```text
Purpose
Typical workflow
Examples
```

例如：

```text
Usage:
  rag_search "keyword"

Purpose:
  Semantic search over internal docs/code/wiki.

Returns:
  Ranked file paths with relevance score.

Typical workflow:
  1. Search topic
  2. Read candidate docs
  3. Refine search if needed

Examples:
  rag_search "bluetooth pairing"
```

---

# 十四、安全设计（重要）

# 不依赖 Prompt 实现安全

Prompt：

```text
Never modify files
```

只是行为建议。

真正安全来自：

# OS 权限

---

# 推荐安全模型

## 1. readonly mount

```yaml
./knowledge:/knowledge:ro
```

---

## 2. Agent 无写权限

Agent 不应：

* 修改知识库
* 修改 embedding 数据
* 修改向量数据库

---

## 3. 最小能力原则

推荐：

```text
Exploration Shell
```

而不是：

```text
Full Bash
```

---

# 十五、推荐 Tool 能力

推荐允许：

* Bash（受限）
* Read
* Grep
* Find
* MCP
* rag_search

不建议：

* unrestricted shell
* root
* 任意写文件

---

# 十六、推荐 Agent Runtime 思路

不要：

```text
自己维护 Agent Loop
```

而应该：

```text
企业工具层
+
成熟 Agent Runtime
```

企业真正的壁垒：

不是：

```text
Agent 本身
```

而是：

```text
企业内部知识 + Tool生态
```

# 十八、最终核心理念

本系统本质：

```text
企业知识探索基础设施
```

而不是：

```text
单纯聊天机器人
```

Agent 负责：

```text
reasoning + orchestration
```

知识系统负责：

```text
retrieval + indexing
```

两者解耦。
